## 1. Цели документа и область применения

### 1.1. Цели

Настоящий документ описывает единые правила кодирования и инженерные практики для разработки и сопровождения систем ИИ-ассистента (далее — «Ассистент»):

- Уменьшить технический долг и хаос в кодовой базе.
- Обеспечить предсказуемость поведения Ассистента.
- Повысить читаемость, поддерживаемость и тестируемость кода.
- Упростить онбординг новых разработчиков.
- Обеспечить единый «голос» и UX-паттерны во всех компонентах Ассистента.

Документ опирается на структуру функционального ТЗ проекта Tips’yo и адаптирован для задач ИИ-ассистента.  

### 1.2. Область применения

Правила распространяются на:

- Бэкенд-сервисы Ассистента (API, оркестрация LLM-запросов, очереди).
- Фронтенд (веб-интерфейс, виджеты, админ-панели).
- Интеграционные компоненты (взаимодействие с внешними API и LLM-провайдерами).
- Инфраструктурные скрипты, CLI-инструменты, крон-задачи.
- Конфигурацию пайплайнов CI/CD, тестов и статического анализа.

## 2. Базовые принципы

1. **Ясность важнее «магии».** Код должен быть понятен человеку без «догадок по интуиции».
2. **Предсказуемость важнее краткости.** Однозначное поведение при одинаковых входных данных.
3. **Безопасность и приватность по умолчанию.** Любые операции с персональными данными реализуются осознанно и минимально.
4. **Fail-fast и прозрачные ошибки.** При критической проблеме — ранний выход с явной ошибкой, а не «тихий» игнор.
5. **Идемпотентность критичных операций.** Повторный запуск не должен приводить к некорректным побочным эффектам.
6. **Трассируемость.** Важные действия и ветки логики должны быть покрыты логами и метриками.
7. **Конфигурация > хардкод.** Всё, что может измениться (ключи, урлы, флаги, лимиты), выносится в конфиги/переменные окружения.
8. **Уважение к пользователю.** Ассистент не должен быть грубым, токсичным, опасным или вводящим в заблуждение.

## 3. Общие требования к структуре проекта

### 3.1. Репозитории

- Каждый крупный компонент Ассистента (core-оркестратор, UI, интеграции) либо:
  - живет в монорепозитории с чёткой модульной структурой, либо
  - оформлен в виде отдельного сервиса с единым шаблоном структуры.
- Минимальный набор директорий (для TypeScript/Python-сервисов):

  ```text
  /src         # основной код
  /tests       # модульные и интеграционные тесты
  /scripts     # утилиты и dev-скрипты
  /config      # конфиги (yaml/json/ts)
  /docs        # локальная документация
  ```
