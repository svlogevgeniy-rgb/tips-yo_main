# Technical specifications — Tips’yo (MVP)

**Документ:** Техническое задание (ТЗ)  
**Продукт:** Tips’yo — цифровая платформа для приёма чаевых в ресторан, кофейня, салон красоты, барбершоп, АЗС, Служба доставки на Бали  
**Версия:** 1.1 (MVP)  
**Дата:** 11.11.2025  
**Ответственные:** Product Owner / Tech Lead / Legal & Compliance

---

## 1. Концепция и цели

**Концепция.** Цифровая платформа для бизнеса, в которой люди работают с людьми: гостям удобнее оставлять чаевые, персоналу — проще их получать.

**Цель проекта.** Разработать сервис для приёма чаевых с поддержкой Xendit → QRIS.

**Ключевая цель MVP.**
- Реализовать приём платежей (чаевых) по QR-коду и по «ссылке на оплату» через Xendit (Payment Links/Hosted Checkout с поддержкой QRIS) и зачисление денег на счета получателей.
- Обеспечить рабочий флоу Xendit → QRIS: создание Payment Link/dynamic QR, обработка webhooks, автоматическое распределение (сплит) долей между официантом и заведением.

**Ключевые показатели успеха MVP:**
- Время от сканирования QR/перехода по ссылке до подтверждения оплаты: ≤ 10 секунд (при нормальной сети).
- Успешная обработка ≥ 99% webhooks без ручного вмешательства.
- Корректный автоматический сплит «по умолчанию» (например, 90/10) с журналированием.

---

## 2. Область охвата (Scope) MVP

Входит в MVP:
1. Приём чаевых через **Xendit Payment Link (Hosted Checkout)** и/или **динамический QR (QRIS)**.
2. Базовый онбординг заведения (кафе/ресторан/салон/АЗС/служба доставки) и сотрудников (официантов и др.): минимальные KYC-поля, привязка банковских реквизитов/идентификаторов для зачисления через PSP.
3. Автоматическое распределение платежа (сплит) между сотрудником и заведением (правило по умолчанию + возможность изменить на уровне сотрудника/смены).
4. Страницы/виджеты:
   - Лёгкий лендинг чаевых (tipsyo.com — тестовый домен), открываемый после сканирования QR/по ссылке.
   - Кабинет заведения: управление сотрудниками, сплит-правилами, операциями и отчётами.
   - Экран благодарности для гостя.
5. Интеграция с Xendit:
   - Создание **Payment Link** (Hosted Checkout) и/или dynamic QR по API.
   - Принятие и валидация webhooks («оплачено», «возврат» — когда доступно).
   - Применение split rule (xenPlatform) при приёме платежа.
6. Хранение и отображение журнала операций, статусов, базовой аналитики.
7. Роли и доступы: Platform Admin, Merchant Admin (Заведение), Staff (Персонал).
8. Локализация: EN/ID (для гостя — автоопределение по браузеру; RU — опционально).
9. Соблюдение MDR QRIS: комиссия PSP не перекладывается на плательщика.

## 3. Определения и сокращения

- **PSP / PJP** — лицензированный платёжный провайдер (в Индонезии).
- **QRIS** — национальный стандарт QR-платежей в Индонезии.
- **Xendit** — платёжный провайдер; используем Payment Links (Hosted Checkout), QR Codes (QRIS), Webhooks, xenPlatform Split.
- **Payment Link (Hosted Checkout)** — хостируемая Xendit платёжная страница (ссылка на оплату).
- **MDR** — Merchant Discount Rate, комиссия эквайринга за приём платежа.
- **Split** — автоматическое деление суммы платежа между несколькими получателями.
- **KYC** — процедуры идентификации заведения/получателей у PSP.

## 4. Пользователи и флоу

### (1) Гость (Payer)

**Как оставить чаевые по QR-коду — флоу (вариант с «ссылкой на оплату»)**
- **Xendit Payment Link (Hosted Checkout):**
  1. Гость сканирует **статический** QR → попадает на лендинг (**tipsyo.com** — тест-домен).
  2. Гость выбирает сумму → бэкенд создаёт **Payment Link** в Xendit. Получаете **URL-ссылку на оплату**.
  3. Отображается кнопка «Оплатить» (ссылка ведёт на Xendit-страницу с методами оплаты, включая **QRIS**).
  4. Гость кликает, выбирает кошелёк/банк и подтверждает платёж.
  5. Xendit шлёт webhook `qr.payment` (или эквивалент события) на наш Callback URL.  

**Сканирование QR-кода (инструкция для гостя)**
1. Откройте камеру телефона и наведите на QR-код.
2. Нажмите на всплывающую ссылку — вы попадёте на страницу оставления чаевых (лендинг Tips’yo).

**Данные карты (прозрачность для гостя)**
1. На странице оплаты вы можете самостоятельно решить, **сохранять карту** для последующих посещений или нет (галочка «Запомнить карту»).
2. Платформа **Tips’yo** (**tipsyo.com** — тест-домен) **не хранит CVV/CVC** вашей карты; реквизиты обрабатываются платёжным провайдером (Xendit/его аутсорс-процессором).

> Примечание: для десктоп-пользователей допускается альтернативный сценарий с динамическим QRIS (сканирование через приложение кошелька), но базовый рекомендованный сценарий — **Payment Link**.

---

### (2) Заведение (Merchant/Кафе)

**Базовые задачи и флоу**
- **Онбординг и KYC:** регистрация, загрузка данных компании (NIB/NPWP/банк-счёт), подключение в Xendit.
- **Сотрудники:** создание карточек сотрудников (ФИО, реквизиты для зачисления), активация/деактивация.
- **Сплиты:** установка правила по умолчанию (например, 90% сотруднику / 10% заведению); опционально — индивидуальные правила на сотрудника/смену.
- **Материалы в зале:** генерация/печать **статического QR** (ведёт на лендинг Tips’yo), размещение на столах/бейджах.
- **Отчётность:** просмотр операций/сумм/статусов, выгрузка CSV, сверка с отчётами Xendit.
- **Комплаенс:** MDR несёт заведение; политика чаевых/налогов — согласно локальным правилам.

**Как гости оставляют чаевые по QR‑коду?**  
1. Гостям не нужно приложение, чтобы оставить чаевые персоналу с помощью платформы.  
2. Достаточно отсканировать QR‑код камерой смартфона или перейти по специальной ссылке.  
3. Ссылка формируется в том случае, если ваш бизнес работает онлайн и нет возможности отсканировать изображение QR‑кода для оставления чаевых.  
4. Дальше — выбор суммы и оставление чаевых. Оставить чаевые можно, используя данные банковской карты и с помощью Xendit/QRIS.  
5. Оставленные чаевые отобразятся в кабинете персонала и будут сразу перечисленные на ваш счёт.  

**Распределение чаевых**  
1. В кабинете заведения Заведение самостоятельно может настроить различные варианты распределения чаевых.  
2. Чаевые могут быть общими: все чаевые идут на один общий аккаунт, выбранный заведением.  
3. Чаевые могут быть индивидуальными: каждый из персонала получает чаевые от гостей, и они отображаются в кабинете персонала.  
4. Также заведение может настроить иное распределение чаевых: например, с каждых оставленных чаевых сотруднику небольшой процент будет уходить другому персоналу — барменам, кухне или др.

---

### (3) Персонал (Staff/Официант)

**Базовые задачи и флоу**
- **Привязка реквизитов:** предоставление данных для зачислений (через заведение, в рамках KYC PSP).
- **Получение чаевых:** доля зачисляется автоматически по сплит-правилу, выбранному заведением.
- **Статистика:** просмотр собственной статистики (кол-во/сумма за смену/период).
- **Прозрачность:** уведомления о новых чаевых (опционально — пуш/e-mail).

## 5. Функциональные требования (уточнённые)

### 5.1. Создание Payment Link (Hosted Checkout) — базовый путь
- POST `/api/tips/intents` (или `/api/payment-links/create`): вход `merchant_id`, `staff_id?`, `amount` (IDR).
- Бэкенд вызывает **Xendit Payment Links API**, получает `payment_link_url` и возвращает его фронтенду.
- Отображается кнопка «Оплатить» → редирект на `payment_link_url`.
- При поддержке сплитов — применить ранее созданное **split rule** (см. 5.3).

### 5.2. (Опционально) Динамический QRIS по API
- POST `/api/tips/intents/qr` → создание **QR Code (QRIS)** через Xendit, возврат `qr_string`.
- Используется преимущественно для десктоп-сценария.

### 5.3. Сплит-платежа (xenPlatform Split)
- Наличие/назначение **Split Rule** (например, 90/10) для заведения/сотрудника.
- Применение `split_rule` к платежу (Payment Link/QRIS) согласно поддерживаемым параметрам Xendit.
- Учёт MDR: оплачивает заведение; Tips’yo не добавляет комиссию плательщику.

### 5.4. Webhooks
- `POST /api/webhooks/xendit` — приём событий (в т.ч. `qr.payment`): валидация `X-CALLBACK-TOKEN`, идемпотентность, апдейт `Tip.status`, запись в `WebhookLog`.

### 5.5. Кабинеты
- **Merchant Admin (Заведение):** сотрудники, сплиты, операции, отчёты, экспорт CSV.
- **Staff (Персонал):** своя статистика и недавние операции.
- **Platform Admin:** заведения, аудит вебхуков, тех.настройки интеграции.

## 7. API (эскиз, дополненный)

### POST `/api/payment-links/create`
**Вход:**  
```json
{ "merchant_id": "m_123", "staff_id": "s_456", "amount": 50000, "locale": "id" }
```
**Логика:** создаёт запись в `Tip` (`status=pending`), вызывает **Xendit Payment Links API**, сохраняет `external_id` и `payment_link_url`, возвращает URL фронтенду.  
**Ответ:**  
```json
{ "external_id": "tip_2025-11-11_s_456_x", "payment_link_url": "https://checkout.xendit.co/..." }
```

### POST `/api/webhooks/xendit`
**Вход:** JSON по схеме Xendit + `X-CALLBACK-TOKEN`.  
**Логика:** валидация, идемпотентность, апдейт `Tip.status=paid`, запись в `WebhookLog`, нотификации.  
**Ответ:** `200 OK`.

### GET `/api/tips/:external_id`
Возвращает `pending|paid|failed|expired` и платежные атрибуты (последний статустаймстамп).

## 8. Данные (схема — основные поля)

**Tip**
- `id`, `external_id`, `merchant_id`, `staff_id|null`, `amount`, `currency=IDR`, `status`,  
  `provider="xendit"`, `xendit_payment_link_url`, `xendit_qr_id|null`, `receipt_id|null`, `created_at`

**Merchant**
- `id`, `name`, `xendit_business_id`, `default_split_rule_id`, `contact_email`, `created_at`

**Staff**
- `id`, `merchant_id`, `name`, `role`, `status`, `xendit_business_id|null`, `payout_ref`, `created_at`

**SplitRule**
- `id`, `merchant_id`, `pct_to_staff`, `pct_to_venue`, `xendit_split_rule_id`, `active`, `created_at`

**WebhookLog**
- `id`, `event_id`, `event_type`, `payload_hash`, `status`, `processed_at`, `raw`
