# Contributing Guide

## 1. Как работать с репозиторием

- Основная ветка — `main`. Для задач создавайте ветки `feature/<topic>`, `fix/<topic>`, `docs/<topic>`.
- Перед началом убедитесь, что локальная ветка синхронизирована: `git pull origin main`.

## 2. Коммиты

- Используйте формат `type: summary`, где `type ∈ {feat, fix, docs, chore}`.
- Один коммит — одна логическая причина изменения.

## 3. Окружение и секреты

- Шаблон переменных — `.env.example` (структура описана в `docs/setup/env.md`).
- Реальные значения хранятся в командном vault (1Password/Bitwarden). Доступ выдаёт тимлид/DevOps.
- Перед запуском создайте локальный `.env`, скопировав значения из vault.
- Любое изменение переменной требует тикета `config/<variable>`, обновления `.env.example` и уведомления в #tipsyo-dev.
- Проверяйте, что `.env` и артефакты не попадают в git (`.gitignore` уже настроен).

## 4. Перед PR

```bash
npm install        # при необходимости
npm run lint
npm run build
npm run preview    # быстрый smoke-тест
```

- Убедитесь, что локально `npm run lint` и `npm run build` проходят без ошибок.
- Если менялись env-переменные, отметьте чек «обновил `.env.example`» в описании PR.
- Не забывайте обновлять документацию (README/docs) при изменении пользовательских сценариев.

## 5. Документация

- Продуктовые изменения — обновляйте `docs/product`.
- UI/UX правила — `docs/ui`.
- Настройки окружения и деплоя — `docs/setup`, `docs/operations`.
- Фиксируйте релевантные изменения в `docs/changelog.md`.

## 6. Code style

- Следуйте `docs/engineering/coding-conventions.md`.
- Используйте существующие UI-компоненты (`src/components/ui`) прежде чем создавать новые.
- Для работы с изображениями придерживайтесь правил из `docs/setup/local.md`.

## 7. CI Pipeline

- Автоматический пайплайн `CI` живёт в `.github/workflows/ci.yml`.
- Он запускается на каждом `pull_request` в `main`, а также на `push` в `main` и `release/**`.
- Шаги пайплайна:
  1. `npm ci`
  2. `npm run lint`
  3. `npm run build`
- PR нельзя мержить, пока пайплайн не зелёный. При падении исправляйте локально и пушьте фикс.

## 8. Code review чек-лист

- ✅ UI соответствует дизайну, нет регрессий в адаптивности.
- ✅ CTA ведут на правильные ссылки.
- ✅ Нет хардкода секретов и чувствительных данных.
- ✅ Документация/README обновлены, если поведение поменялось.
- ✅ `.env.example` обновлён, если добавлялись новые переменные.
- ✅ CI пайплайн `CI` зелёный и не содержит предупреждений уровня error.
